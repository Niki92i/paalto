03 · how it behaves

Six properties, not features.

01 · AGNOSTIC

Your data, your stack, your models.

Works with whatever repo, CRM, warehouse, or model you already run. Not locked to any vendor, including us.

02 · INTEROPERABLE

Talks to everything with an API.

GitHub, Linear, Notion, Slack on day one. Figma, Posthog, Jira shipped. Anything with an MCP plugs in.

03 · INTELLIGENT

Frontier models under the hood.

Claude by default. Updated as new models ship. The system inherits their intelligence directly.

04 · COMPOUNDING

Every run sharpens the next.

Every action lands in runs/<id> as a transcript and structured events. What worked becomes default.

05 · MODIFIABLE

Markdown configs, code you own.

Skills, agents, prompts — all .md files. Fork it, extend it, prune it. No magic. No lock-in.

06 · HEADLESS

Chat is the interface.

No dashboard to fight. Conversation is how you operate it. The transcript is your audit trail.

05 · questions

Questions.

Do I need to be technical?
No. You drive paalto in plain English in Claude Code. If you can describe the outcome, the system can run it. The Engineer agent handles the code.
Where does my data live?
On your machine. paalto runs locally. Your transcripts, PRDs, and keys never go to a paalto backend — there isn't one. Model API calls go directly from your machine to your model provider.
Will it auto-merge my code?
No. Every PR opens as draft. The PR Builder skill is hard-coded to refuse merge calls. Merge approval is a named human gate.
What's the difference between paid and OSS?
Time and friction. The OSS repo is free and documented; getting it fully wired into your stack takes a few hours of setup. The paid implementation compresses that into a single session with the team that runs paalto every day.
Can I add my own skills?
Yes — that's the whole point. A skill is a single .md file. See CONTRIBUTING.md.
